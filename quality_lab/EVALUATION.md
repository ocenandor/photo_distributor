# Quality Lab Evaluation

This document describes how to compare face-analysis experiments in the local
quality lab.

The fixed lab baselines live in `quality_lab/EXPERIMENT_BASELINES.md` and
`quality_lab/experiment_baselines.json`. They are intentionally not production
workflow defaults.

## Inputs

Each evaluation uses two files:

- `quality_lab/data/labels.json` - manual labels.
- `quality_lab/data/runs/<run_id>/predictions.json` - output of one algorithm
  run.

`predictions.json` also declares which people have reference embeddings in that
run. This matters because a person can be labeled in the dataset but impossible
to recognize until their reference photos are added.

## Decision Model

For every detected face the evaluator creates a face decision:

- `correct`: known reference person was accepted correctly.
- `missed_known_person`: known reference person was below threshold.
- `wrong_identity`: known reference person matched as someone else.
- `rejected_no_reference`: real subject has no reference in the run.
- `false_accept_no_reference`: real subject without a reference was accepted as
  a known person.
- `rejected_irrelevant`: background or non-subject face was rejected.
- `false_accept_irrelevant`: background or non-subject face was accepted.
- `rejected_nonface`: false positive detection was rejected.
- `false_accept_nonface`: false positive detection was accepted.
- `unlabeled_prediction`: prediction has no manual label yet.

For every photo the evaluator creates a distribution decision:

- expected recipients are `photo_subjects` filtered to people with references;
- predicted recipients are people whose best face match is above the threshold;
- false recipients are predicted people who should not receive the photo;
- missed recipients are expected people who did not receive the photo;
- quarantine is correct when both expected and predicted recipient sets are
  empty for the current reference set.

## Metric Layers

Dataset coverage:

- total images;
- labeled images;
- predicted faces;
- labeled faces;
- unlabeled predictions.

Detection quality:

- real faces;
- false non-face detections;
- real but irrelevant background faces;
- distribution-relevant faces;
- subject-label coverage.

Detector noise is not automatically a severe product issue. A false or
background detection becomes severe when it is accepted as a known person and
causes a wrong photo copy. If the matcher rejects it, it is mostly a quality and
performance concern.

Recognition quality:

- recall on people with references;
- accepted-face precision;
- rejection rate for people without references;
- rejection rate for noise faces.

Reference robustness:

- per-person subject-face recall;
- score distributions against that person's own references;
- misses grouped by occlusion, pose, and quality;
- hard cases where the detector found a valid subject face, but the embedding
  score stayed below the matching threshold.

Multi-reference enrollment should be treated as an internal quality mechanism,
not as a product requirement. A participant may upload one reference photo. The
lab can still compare several reference embeddings for the same person when
they are available, or when we later add safe internal enrollment strategies.
The important diagnostic question is which reference image helps or hurts each
hard subject photo.

The "single person in photo, give it to the nearest participant" rule is useful
as an experiment, but it should not be an automatic high-confidence decision by
itself. If every reference score is very low, the nearest person can still be
meaningless. In that case the safer state is `candidate_review`, not direct
copying.

Distribution quality:

- recipient precision;
- recipient recall;
- recipient F1;
- exact-photo rate;
- quarantine accuracy.

## Generated Artifacts

Run:

```powershell
.\.venv\Scripts\python.exe scripts\quality_lab_metrics.py --run-id interactive_baseline_0_6
```

The evaluator writes:

- `metrics.json` - nested summary metrics for the selected threshold.
- `face_decisions.csv` - one row per detected face.
- `photo_decisions.csv` - one row per photo.
- `reference_cases.csv` - one row per relevant subject face, focused on
  reference matching.
- `reference_slices.csv` - aggregated reference robustness by person,
  occlusion, pose, and quality.
- `reference_image_diagnostics.csv` - one row per reference embedding with
  basic reference-image quality checks.
- `reference_subject_diagnostics.csv` - one row per relevant subject face with
  its best same-person reference score.
- `reference_score_details.csv` - one row per subject face per same-person
  reference embedding, useful for comparing multiple references.
- `reference_subject_slices.csv` - reference robustness slices generated by the
  dedicated reference diagnostics script.
- `threshold_sweep.csv` - metric comparison across matching thresholds.

Reference diagnostics:

```powershell
.\.venv\Scripts\python.exe scripts\quality_lab_reference_diagnostics.py --run-id interactive_baseline_0_6
```

Derived-reference diagnostics:

```powershell
.\.venv\Scripts\python.exe scripts\quality_lab_derived_references.py --source-run-id sface_baseline --run-id sface_derived_strict_0_65 --min-score 0.65
```

The derived-reference script writes a normal second-pass `predictions.json` plus:

- `derived_reference_selection_audit.csv` - every first-pass face and why it was
  selected or rejected as a derived reference;
- `derived_reference_candidates.csv` - selected derived references with manual
  label risk columns.

When comparing derived runs, look at both the default threshold metrics and the
`threshold_sweep.csv`. Derived references may not change quality at the original
threshold but can make a lower second-pass threshold safer.

The currently fixed derived-reference lab baseline is:

```powershell
.\.venv\Scripts\python.exe scripts\quality_lab_derived_references.py --source-run-id sface_baseline --run-id sface_derived_0_50_det_0_75 --min-score 0.50 --min-detection-score 0.75 --max-per-person 4
.\.venv\Scripts\python.exe scripts\quality_lab_metrics.py --run-id sface_derived_0_50_det_0_75 --match-threshold 0.35
```

Do not transfer these settings to production until the dataset includes more
events and people.

Model comparison:

```powershell
.\.venv\Scripts\python.exe scripts\quality_lab_compare_runs.py
```

Backends should write the same `predictions.json` shape. The default `sface`
backend uses OpenCV YuNet + SFace. The optional `insightface` backend uses
InsightFace model packs and can run on CPU or CUDA when the matching runtime is
installed.

## Current Baseline Notes

The first labeled dataset has one active reference person.
Other labeled people are useful for false-accept and relevance tests, but they
cannot be recognized correctly until their references are added.

On the current `interactive_baseline_0_6` run with match threshold `0.45`, the
main failure mode is recall: hard photos with mask/helmet/reflection or very
dark profile lighting are missed. Lowering the threshold to about `0.30` helps
on this dataset without adding false recipients, but this needs more labeled
photos before becoming a production default.

The useful question for the current baseline is not just "did YuNet detect a
box?" It is "did the detected subject face join to the correct references?"
For example, non-face/background detections that do not match a referenced
person are not a distribution error. The important current errors are valid
referenced-person subject
photos in sunglasses, mask, helmet, reflection, dark light, or profile pose
that fall below the matching threshold and go to quarantine.
